# PROJECT GOAL

Build an application that:

1. Accepts a `.docx` file upload
2. Extracts its content into structured sentence-level units
3. Sends sentences in chunks to an LLM API for proofreading
4. The LLM identifies:
   - grammar errors
   - typos
   - incorrect word usage
   - unnatural phrasing (ONLY if clearly non-native or confusing)
5. The LLM returns STRICT structured JSON corrections only (no prose rewriting of full paragraphs)
6. The backend maps corrections back to sentence IDs
7. The system injects comments into the original Word document next to the relevant sentences
8. Outputs a new `.docx` file with comments
9. User downloads annotated document

# CRITICAL DESIGN CONSTRAINTS

## 1. NO FULL TEXT REWRITES

The LLM must NEVER rewrite full sentences or paragraphs.

Only allowed outputs:
- single-word corrections
- minimal phrase replacements
- short suggestion lists (max 3 options) ONLY for unnatural phrasing

# 2. SENTENCE ID SYSTEM (MANDATORY)

All text must be preprocessed into structured sentences.

Each sentence MUST have:
- paragraph_id (int)
- sentence_id (int)
- text (string)

Example:

{
  "paragraph_id": 3,
  "sentence_id": 18,
  "text": "She go to school every day."
}

The LLM MUST reference sentence_id only.

No quoting of original sentences is allowed.

# 3. CHUNKING STRATEGY

- Default chunk size: full chapter (~2000 words)
- If chapter exceeds token limit:
  - split by paragraph boundaries first
  - then by sentence grouping
- Maintain sentence IDs across chunks
- Preserve ordering

# 4. LLM OUTPUT FORMAT (STRICT JSON ONLY)

The LLM must output ONLY valid JSON in this schema:

{
  "issues": [
    {
      "sentence_id": 12,
      "type": "GRAMMAR | TYPO | TENSE | PREPOSITION | WORD_CHOICE | UNNATURAL",
      "issue": "short description of problem",
      "fix": "minimal correction OR corrected token/phrase",
      "suggestions": ["optional", "only", "for unnatural phrasing"]
    }
  ]
}

Rules:
- If TYPO or GRAMMAR → only ONE fix allowed
- If UNNATURAL → max 3 suggestions
- No paragraph rewrites
- No stylistic rewriting
- No explanations unless necessary for UNNATURAL type
- Must be machine-parseable JSON

# 5. LLM DETECTION RULES

The model must ONLY flag:

Allowed:
- grammatical errors
- verb tense errors
- incorrect prepositions
- spelling mistakes
- incorrect word usage
- phrases that would clearly sound non-native or confusing to a fluent English reader

Forbidden:
- stylistic improvement
- rewriting for elegance
- improving tone
- changing voice
- removing fragments that are intentional
- subjective “better phrasing”

# 6. OUTPUT MINIMIZATION RULE

To reduce token usage:

- TYPO → only corrected word
- GRAMMAR → only corrected phrase
- WORD_CHOICE → minimal replacement
- UNNATURAL → max 3 alternatives, no explanation unless necessary

## D. CORRECTION MAPPER

## E. DOCX ANNOTATOR
- Inserts Word comments at sentence level

## F. EXPORT MODULE
